# GroceryView data architecture

This page is the data-focused companion to [`docs/ARCHITECTURE.md`](ARCHITECTURE.md). It exists so database, ingestion, and generated-artifact details are reachable from the top-level architecture map without overloading that overview.

## Authoritative data stores

| Store | Main paths | Purpose |
| --- | --- | --- |
| PostgreSQL / Prisma | `packages/db`, `prisma/schema.prisma`, `infra/db` | Normalized catalog, chains, stores, source runs, raw records, observations, latest prices, preferences, and operational state. |
| Generated web modules | `apps/web/src/lib/ingested`, `apps/web/src/lib/generated` | Static snapshots used by the web app when builds or runtime routes need checked-in or exported evidence. |
| Cached public assets | `packages/image-cache` | Product/store imagery and derived cache metadata with source provenance. |
| Workflow artifacts | `.github/workflows`, `docs/ops/production-daily-ingestion-readiness.md` | JSON diagnostics for production config, migrations, ingestion runs, DB snapshots, and readiness probes. |

## Data lifecycle

1. Connector configuration and store enumeration are generated by `scripts/ops` and `packages/ingestion`.
2. Ingestion fetches public retailer, aggregator, store, fuel, OSM, Open Food Facts/OpenPrices, or receipt-derived evidence.
3. The DB-backed runner persists source-run status, raw payload provenance, normalized observations, and latest-price rows.
4. Export scripts create web-safe generated modules from public normalized rows and coverage summaries.
5. Readiness endpoints and workflows validate freshness, required chain/store/product/category coverage, price-type coverage, and stale-observation blockers.
6. The web app and API expose only normalized rows and confidence/provenance labels, not raw private payloads.

## Design rules

- Keep source provenance: every observation should retain source URL, source run, retrieval time, and store/chain identifiers when available.
- Fail closed on empty or under-threshold captures; a partial scrape is a blocker, not a success signal.
- Separate retailer-native ids from GroceryView canonical ids until matching has explicit evidence.
- Generated modules are artifacts; regenerate them from scripts or DB exports unless a ticket explicitly asks for fixture edits.
- Coverage gates must prove the exact launch dimensions they claim: chain, store, product, category, price type, and freshness.
